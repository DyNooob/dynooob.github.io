最近有人把 Claude Design 的系统提示词从仓库里扒了出来，647 行，开源在 GitHub 上。我看了一遍，发现它把 AI 做设计为什么总是一股「AI 味儿」这件事说得挺清楚。

## 身份定位

提示词开头第一句就定调了：

> You are not a code generator who happens to make designs. You are a designer who happens to use code.

这不是文字游戏。它区分了两种工作方式：

- 生成代码的 AI 会输出看起来还不错的页面，把内容填满就完事
- 设计师会先问这一页到底要干什么、用户第一眼该注意哪里、哪些可以直接干掉

三层具体要求：
1. 先确定颜色、字体、间距、组件等规则，整套设计按规则来，而不是逐页拼凑
2. 用户说加一个模块时，如果加东西会把设计搞崩，应该解释原因并提出反对意见
3. 把用户当 manager，保持专业判断力但也尊重最终决策权

## 工作流：先搭骨架，尽早展示

提示词对 AI 的提问行为做了约束：只问那些答案有可能会改变设计方向的问题。改个按钮样式、调个默认值、做两种差不多的排版——AI 自己决定就行，交付时提一句即可。

有一条很实用：Build a skeleton, show it early。先把框架搭出来，尽早展示。最怕的是在后台吭哧吭哧做完 15 页，然后发现方向一开始就错了。

## 反「AI 味儿」清单

这是全文最直接的部分。提示词给 AI 列了一份黑名单：

- **三色以上大渐变**：默认别用，彩虹霓虹色就是 AI 味模板
- **Emoji 标题**：标题前面硬加 🚀、📈、✅ 直接干掉
- **过度卡片化**：普通内容不需要 `border-radius: 12px` + `border-left: 4px solid`，左边状态色条应该留给提醒和警告
- **奶油底 + 衬线大标题 + 陶土色**：最近流行的这套组合被点名了，用在杂志或酒店网站没问题，但别往开发者工具、企业后台、金融医疗领域套

## 视觉结构：间距和字体的纪律

提示词要求间距从 4px 或 8px 的倍数里选：8、16、24、32。字体大小也固定：正文 16px，说明文字 14px，标题从 24、30、36 里选。不允许出现 19px、21px 这种零散值。

视觉层次的要求更具体：标题、说明、按钮如果差不多大，用户的关注点就散了。把标题字号拉开，只留一个主按钮，层次感就出来了。

## 交互状态：五种状态是底线

一个按钮至少准备五种状态：default、hover、active、focus、disabled。如果需要发送请求，还要加上 loading。

页面上最重要的按钮只能有一个。AI 很容易把页面做成按钮展览——开始使用、预约演示、查看文档、了解价格，全都做得又大又亮。用户看完根本不知道该点哪个。

## 组件化思维

原文有一句：Design components, not pages。

先把能复用的组件做好，再用组件去填充页面。一个产品页可以拆成导航、首屏、功能卡片、行动按钮、底部信息。按钮要有 primary、secondary、ghost 三种变体，配合尺寸、图标和 loading 状态。间距、颜色、字体、圆角、阴影都应该能复用。

## 功能优先级

用户说要做五个功能，AI 很容易先搭五个空壳。提示词要求先找出最重要的三个，把入口、操作、错误状态和最终结果全部做完。比如结账功能：用户要能填写地址、选择付款方式、提交订单、看到 loading、看到错误提示、看到成功或失败的结果页。

优惠券、会员积分、邀请好友等功能不着急加。

## 双 Agent 检查机制

每次完成明显的视觉修改后，一个专门的 verifier agent 会接手检查。它不负责重新设计，只做三件事：截图检查布局、用键盘走一遍页面、检查 JavaScript 报错。发现输入框没有 label、按钮对比度太低、focus 看不见，就抛出来让主 agent 改。

改完之后还有一个 polish-pass 阶段，调动四个检查 agent 分别检查视觉阻碍、AI 模板味、页面层级和交互状态。

## 设计流程的两种路径

从零开始做页面：
```
discovery-questions → frontend-aesthetic-direction
→ wireframe → make-a-prototype → polish-pass
```

已有品牌和现成页面：
```
design-system-extract → generate-variations
→ make-tweakable → polish-pass
```

## 几点观察

这份提示词的本质是把「设计品味」写成了可执行的规则。它不是让 AI 自由发挥，而是给了一整套判断标准——什么该做、什么不该做、做到什么程度算完。

647 行看起来很多，但大部分是边界条件和异常处理。核心逻辑并不复杂：先定方向、搭骨架、做组件、填内容、检查、修改。这个流程本身不新鲜，新鲜的是把它写成了 AI 能执行的形式。

另一个有意思的点是双 Agent 设计。生成和检查分离，让一个 agent 专门挑另一个的错。这在复杂任务中比让同一个 agent 自检有效得多。
