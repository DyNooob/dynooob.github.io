Claude Design 的系统提示词（共 647 行）在 GitHub 上公开后，引起了提示工程和 AI 辅助设计领域的关注。这份提示词的价值不在于它的长度，而在于它系统性地将设计规范转化为 AI 可执行的指令集。以下从提示工程和设计系统的角度进行分析。

## 身份锚定：AI 的角色定位策略

提示词开篇即建立了明确的角色锚定：

> You are not a code generator who happens to make designs. You are a designer who happens to use code.

这是一个典型的提示工程技巧——通过否定-肯定结构（"不是 X，而是 Y"）来压缩 AI 的推理空间。代码生成器与设计师的区别在于：前者倾向于填充内容，后者优先理解需求再选择表达方式。这种身份锚定直接影响了后续所有决策的走向。

提示词进一步分解了角色期望的三个层次：建立设计规则体系（而非逐页拼凑）、在用户需求破坏设计系统时主动提出异议、以专业判断力辅助决策但尊重最终决定权。

## 工作流约束：减少无意义交互

提示词对 AI 的提问行为做出了明确的约束——只提那些答案可能改变设计方向的问题。样式细节、默认值选择、排版微调等多由 AI 自主决定，仅需在交付时附带说明。这是减少交互轮次、提升效率的关键设计。

工作流的核心逻辑是"先搭骨架，尽早展示"（Build a skeleton, show it early）。这源于一个常见问题：AI 在后台生成完整页面集后发现方向错误，造成大量无效工作。骨架先行策略将反馈周期前置，降低了试错成本。

## 设计系统的规则化表达

提示词从多个维度构建了可执行的设计规则：

### 反模式清单
提示词明确列出了 AI 在设计中的常见问题：三色以上渐变色（默认禁用）、标题前硬插 Emoji、过度使用圆角卡片和左侧状态色条、以及"奶油底色 + 衬线大标题 + 陶土色"的流行组合——后者在特定场景（如酒店、杂志网站）中合理，但不适用于开发者工具、企业后台或金融医疗类产品。

### 间距与字体的约束
采用 4px/8px 倍数的间距系统（8、16、24、32），字体大小固定在 16px（正文）、14px（说明文字）、24/30/36px（标题），不允许出现 19px、21px 等随机值。这是设计系统 token 化的体现，通过约束取值范围来保证视觉一致性。

### 交互状态的最小完备集
每个按钮至少需要五种状态：default、hover、active、focus、disabled，发送请求时增加 loading 状态。页面上仅保留一个主按钮，避免多个高亮按钮造成用户决策负担。

### 组件化优先
"Design components, not pages" 是提示词的核心原则之一。可复用的组件（导航、首屏、功能卡片、行动按钮、底部信息）先构建完成，再用组件填充页面。按钮需提供 primary、secondary、ghost 三种变体，配合尺寸、图标和 loading 状态，通过 token 化的间距、颜色、字体、圆角、阴影实现复用。

### 功能优先级的分层实现
提示词要求识别出最核心的三个功能，将其入口、操作、错误状态和最终结果完整实现，而非为所有功能搭建空壳。例如结账功能需完整覆盖地址填写、付款方式选择、订单提交、loading 状态、错误提示、成功/失败结果页。优惠券、会员积分、邀请好友等功能延后实现。

## 双 Agent 质量保证机制

提示词中引入了一个专门的 verifier agent，在每次视觉修改后执行三项检查：截图检查布局、键盘走查页面、检查 JavaScript 错误。它不负责重新设计，仅负责发现问题并反馈给主 agent 修改。

在 polish-pass 阶段，四个检查 agent 分别关注视觉阻碍、AI 模板味、页面层级和交互状态。这种生成与检查分离的架构，在复杂任务中比单一 agent 自检更有效——两个独立 agent 的交叉验证降低了自我盲区。

## 设计流程的两种模式

提示词定义了两条路径：

- 从零开始：discovery-questions → frontend-aesthetic-direction → wireframe → make-a-prototype → polish-pass
- 已有品牌/页面：design-system-extract → generate-variations → make-tweakable → polish-pass

前者侧重需求探索和方向确定，后者侧重从现有设计中提取可复用的系统规则并生成变体。

## 设计系统的可执行化

从提示工程的角度看，这份提示词的本质是将"设计品味"转化为可判定的规则集。647 行中大部分是边界条件和异常处理，核心逻辑并不复杂：定方向、搭骨架、做组件、填内容、检查、修改。真正的创新在于将人类设计师的隐性知识——什么该做、什么不该做、做到什么程度算完成——编码为 AI 可执行的显性规则。这对于设计系统在 AI 辅助场景下的落地具有参考价值。
