API 是现代应用的骨架。前后端通信靠 API，服务间调用靠 API，AI Agent 的交互也靠 API。但 API 也是最容易出安全问题的环节——OWASP 的 API Security Top 10 每年都在更新，而实际测试中暴露的问题往往集中在几个固定模式上。

以下是我整理的在 API 安全测试中值得优先检查的 10 个点。

## 1. 认证绕过

检查 API 端点是否真的需要认证。最简单的测试：拿掉请求头中的 Authorization 或 Cookie，看接口是否还能返回数据。

很多 API 只在文档里写了「需要认证」，但实际实现中部分端点缺少验证。尤其是文件上传、数据导出、配置查询这类容易被忽略的接口。

## 2. 越权访问

通过认证后，测试能否访问其他用户的资源。典型的测试方法：

- 用用户 A 的 token 访问用户 B 的资源（IDOR）
- 普通用户尝试访问管理端接口
- 修改请求中的用户 ID、订单号等参数

RESTful API 中 `/api/user/123/orders` 这样的模式最容易出现越权——如果后端只校验了 token 有效性，没校验 token 所属用户与请求资源的 ID 是否匹配，就会出问题。

## 3. 输入注入

所有接受用户输入的参数都可能成为注入点。SQL 注入在传统 Web 应用中已大幅减少，但 NoSQL 注入、命令注入、LDAP 注入在 API 场景中仍然常见。

特别关注 JSON 格式的请求体——某些框架会递归解析嵌套对象，可能导致意外的参数覆盖或注入。

## 4. 频率限制

检查关键接口是否有频率限制。登录、注册、密码重置、短信验证码、支付等接口如果没有限流，会被暴力破解或滥用。

测试方法：短时间内连续发送请求，观察是否触发限流机制。如果 100 次请求全部成功返回，说明没有限流。

## 5. 敏感信息泄露

API 响应中常见的泄露：

- 错误堆栈信息直接返回给前端
- 数据库表结构、SQL 语句在错误信息中暴露
- 密码哈希、Token、API Key 在响应中返回
- 调试接口在生产环境未关闭

检查所有错误响应，看看是不是返回了比「500 Internal Server Error」更多的信息。

## 6. 批量赋值

API 端点接受用户输入后直接绑定到数据模型，可能导致用户修改不该修改的字段。

典型案例：用户注册接口只应该接受 username 和 password，但后端直接绑定到 User 模型，用户可以在请求中附加 `"role": "admin"` 来提升权限。

测试方法：在请求中添加额外的字段，观察是否被写入数据库。

## 7. 分页与遍历

不设限的分页接口可能导致数据泄露。如果 `/api/users?page=1` 返回所有用户信息，攻击者可以通过遍历 page 参数获取全量数据。

测试方法：检查分页接口是否有上限限制，是否对敏感数据做了访问控制。

## 8. 文件上传

文件上传接口是最容易出问题的端点之一。检查项：

- 文件类型校验是否可靠（仅检查扩展名是不够的）
- 文件大小限制
- 上传路径是否可预测
- 上传的文件是否可直接访问

## 9. 会话管理

- Token 是否在 URL 中传递（应该只在 Header 中）
- Token 是否设置了合理的过期时间
- 登出后 Token 是否立即失效
- 密码修改后旧的 Token 是否失效

## 10. Content-Type 与序列化

检查 API 是否正确验证了 Content-Type 头。如果接口期望 JSON 但接受 `application/x-www-form-urlencoded`，可能导致参数解析差异，绕过安全校验。

## 测试工具

- curl：快速测试单个端点
- Burp Suite：拦截和修改请求
- Postman：组织和自动化测试用例
- 自定义脚本：批量测试和参数遍历

## 优先级建议

如果资源有限，优先测 1、2、6——认证绕过、越权访问和批量赋值是最容易出严重漏洞的环节，而且测试成本低，不需要复杂的工具配置。
