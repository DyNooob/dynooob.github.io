## 分块是 RAG 的第一个关键步骤

文档切分（Chunking）看似简单，实际上直接影响检索质量。块太大，信息混杂；块太小，上下文不足。怎么切分最好？没有标准答案，取决于你的文档类型和应用场景。

## 常见分块策略

### 1. 固定长度分块

最简单的方案，按字符数或 token 数切分。

```
[第1-500字] [第401-900字] [第801-1300字]
      重叠 100 字
```

**优点：** 实现简单，速度快
**缺点：** 可能在句子中间切断，破坏语义

**适用场景：** 快速原型，文档结构不明显

### 2. 递归分块

按段落 → 句子 → 子句的优先级切分，尽量在自然边界断开。

```
## 第一章            ← 一级标题
  1.1 背景          ← 二级标题
    第一段内容...    ← 段落
    第二段内容...
  1.2 方法
    第一段...
```

LangChain 的 `RecursiveCharacterTextSplitter` 就是这种思路。

**优点：** 保留语义完整性
**缺点：** 实现较复杂

### 3. 语义分块

用 Embedding 模型检测文本的语义边界，语义变化处就是切分点。

```
段落 A（讲背景） → 语义边界 → 段落 B（讲方法）
```

**优点：** 切分最合理
**缺点：** 需要调用 Embedding 模型，耗时较长

### 4. 结构化分块

针对特定格式的文档进行智能切分。

| 文档类型 | 切分策略 |
|----------|----------|
| Markdown | 按标题层级切分 |
| HTML | 按 DOM 节点切分 |
| PDF | 按章节/段落切分 |
| 代码 | 按函数/类切分 |

## 块大小怎么选？

| 场景 | 推荐块大小（token） | 原因 |
|------|-------------------|------|
| 问答（事实型） | 200-500 | 答案通常在一段内 |
| 摘要 | 1000-2000 | 需要完整上下文 |
| 代码检索 | 100-300 | 一个函数通常很小 |
| 论文分析 | 500-1000 | 段落级信息 |

## 重叠（Overlap）的重要性

块之间设置重叠可以避免在关键位置切断。

```
[第1-500字]
     [第401-900字]
          [第801-1300字]
```

重叠量通常设为块大小的 10%-20%。

## 实际建议

1. **先用固定长度试**。快速出效果，再优化。
2. **Markdown/HTML 文档优先用结构化分块**。保留标题层级能大幅提升检索命中率。
3. **块大小不是越小越好**。太小的块丢失上下文，太大的块引入噪声。
4. **多测试不同的重叠率**。10%-20% 是经验值，具体看文档结构。

> **一句话：** 没有万能的分块策略，根据你的文档类型和查询特点调整才是最有效的做法。
