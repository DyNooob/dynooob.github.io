## 大模型的痛点

大语言模型（LLM）有一个天然缺陷：**知识截止于训练数据的时间点**。比如问它今年发生的事，它不知道。而且它可能 **"一本正经地胡说八道"**（幻觉）。

RAG 就是来解决这个问题的。

## 什么是 RAG？

**RAG = Retrieval-Augmented Generation（检索增强生成）**

简单说：**让 AI 先去查资料，再根据查到的资料回答。**

```
用户提问
    ↓
┌─────────────┐
│  检索知识库  │  ← 从文档/数据库/网页中找出相关内容
└─────────────┘
    ↓
┌─────────────┐
│  LLM 生成   │  ← 基于检索到的内容 + 用户问题，生成答案
└─────────────┘
    ↓
  带引用的回答
```

## RAG 的核心流程

### 1. 文档切分
把长文档切成小块（chunk），每块几百到几千字。

### 2. 向量化
用 Embedding 模型把每块文本转成向量（一组数字）。

### 3. 存储
把向量存入向量数据库（如 Chroma、Pinecone、Milvus）。

### 4. 检索
用户提问时，把问题也转成向量，在数据库中找最相似的 chunk。

### 5. 增强
把检索到的 chunk 拼到 prompt 里，作为上下文。

### 6. 生成
LLM 根据上下文 + 问题，生成带引用的答案。

## RAG vs 长上下文

2024-2025 年，模型上下文窗口从 8K 扩展到 128K、200K 甚至 1M token。那还需要 RAG 吗？

**需要。** 原因：
- **成本**：1M token 的输入比检索贵得多
- **速度**：检索几毫秒，处理 1M token 要几十秒
- **精度**：检索能精准定位相关信息，长上下文可能淹没在噪声中

## RAG 的典型应用

| 应用 | 检索源 | 效果 |
|------|--------|------|
| 客服机器人 | 产品文档 | 准确回答产品问题 |
| 法律助手 | 法规库 | 引用具体法条 |
| 代码问答 | API 文档 | 给出最新 API 用法 |
| 论文阅读 | 论文库 | 对比多篇论文观点 |

> **一句话总结：** RAG = 给 LLM 配一个外部知识库，让它先查再答，既解决了知识截止问题，又减少了幻觉，还能给出引用来源。
