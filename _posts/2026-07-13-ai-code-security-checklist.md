使用 AI 辅助生成代码的开发模式正在快速普及。Cursor、Claude、GPT 等工具大幅缩短了从需求到上线的周期，但 AI 生成代码的安全隐患也随之增加。以下从安全开发流程（SDL）的角度，整理了一份上线前的安全检查清单，按风险等级从高到低排列。

## 第一级：法律合规（最高优先级）

### 1. 用户数据收集合规审查
只要应用收集任何用户数据——包括邮箱注册、用户行为日志、页面访问记录——相关法律法规即已适用。在中国适用的法律包括《个人信息保护法》（PIPL）和《数据安全法》，在欧盟运营需遵守 GDPR，在美国需关注 CCPA 等州级法规。合规检查的核心在于：确认是否已制定并公开隐私政策，且隐私政策的内容与实际数据收集行为一致。AI 生成的应用容易忽略这一层——后端可能没有数据收集逻辑，但前端表单、第三方 SDK 或分析工具可能已经在未经声明的情况下收集数据。

### 2. 数据存储位置与权限审计
需明确回答以下问题：用户数据存储在哪个服务器、哪个地区？哪些人员或系统有访问权限？数据在传输和存储过程中是否加密？存储方式是否符合适用法规的数据本地化要求？AI 生成项目时倾向于选择默认配置，而默认配置往往不会自动满足这些合规要求。

## 第二级：安全基线（高优先级）

### 3. 安全响应头配置
安全头部的配置是投入产出比最高的安全措施。需检查以下头部是否已正确配置：Content-Security-Policy（CSP）、X-Frame-Options、HTTP Strict-Transport-Security（HSTS）、X-Content-Type-Options、Referrer-Policy 等。AI 生成的代码通常不会主动添加这些头部，需要在部署配置中显式补充。

### 4. OWASP 标准安全审查
建议在开发流程中嵌入 OWASP Top 10 作为标准审查项。AI 生成代码中常见的漏洞包括：SQL 注入（字符串拼接查询）、跨站脚本（XSS，dangerouslySetInnerHTML 等不安全渲染方式）、认证绕过、不安全的反序列化等。这些漏洞与人类开发者犯下的错误类型相同，但 AI 更容易在无意识状态下重复这些模式，尤其在缺乏安全上下文提示的情况下。

### 5. 注入与认证专项检查
AI 生成的 SQL 查询倾向于使用字符串拼接而非参数化查询，前端代码容易出现直接渲染用户输入的情况。需专项检查所有数据库操作是否使用参数化查询或 ORM 的安全方法，所有用户输入是否经过充分的验证和转义，以及认证流程是否存在绕过路径。

### 6. 环境变量隔离验证
AI 在生成前端代码时有时会将环境变量引用混入客户端代码。检查方法：在构建产物中搜索所有环境变量引用，确认任何 API key、数据库连接字符串、密钥等敏感信息均未出现在前端代码中。任何在浏览器端可访问的变量都应视为公开信息。

## 第三级：数据泄露防护（中高优先级）

### 7. API 响应数据最小化
AI 编写的 API 接口有一个常见模式：直接返回完整的数据表对象（如 `return user`），而前端只使用了其中少数几个字段。这会导致用户信息在 API 响应中过量暴露。修复方法：为每个 API 接口定义明确的数据传输对象（DTO）或显式筛选返回字段，遵循"最小必要"原则。

### 8. 日志中的敏感信息清理
调试阶段植入的 `console.log(apiKey)`、`print(secret)` 等调试语句进入生产环境后，可能通过日志系统泄露到第三方监控服务。需在代码库中搜索所有日志输出语句，确认其中不包含密钥、Token、用户个人信息等敏感数据。建议在 CI/CD 流程中集成日志内容扫描。

### 9. 前端密钥的安全假设
浏览器端出现的任何密钥都应视为已泄露。攻击者通过 DevTools 的 Network 面板即可在数秒内获取所有前端可见的密钥。因此，所有第三方 API 调用应由后端服务完成：前端将请求发送到自身后端，后端代为调用第三方服务，第三方 API key 仅存在于服务端环境变量中。

## 第四级：运营安全防护

### 10. 密钥管理重构
所有密钥必须从代码库中移除，通过环境变量、密钥管理服务（如 AWS Secrets Manager、HashiCorp Vault）或配置中心注入。AI 生成代码时倾向于将密钥直接写在配置文件或初始化代码中，需要在安全审查中逐一排查并迁移。

### 11. 频率限制（Rate Limiting）配置
AI 生成的应用很少包含频率限制。缺少频率限制的后端接口面临的风险包括：API 账单耗尽攻击、暴力破解、爬虫抓取、DDoS 等。建议在应用层或网关层为所有对外接口配置基于 IP 或用户维度的频率限制策略，并配合监控告警。

## 安全审查流程集成

上述检查点不应作为上线前的单次突击检查，而应嵌入到安全开发流程中。建议的集成方式：

1. 在 AI 生成代码后，使用安全审查 prompt（如"Review my app against OWASP standards and highlight vulnerabilities"）对代码进行自动化扫描
2. 将安全头部检查、密钥泄露检测、日志清理等可自动化项集成到 CI/CD 流水线
3. 将法律合规、数据存储审计等需要人工判断的项纳入上线审批流程

AI 生成的代码带来了效率提升，也引入了新的安全风险维度。通过系统化的安全检查流程，可以在享受效率优势的同时控制风险敞口。
